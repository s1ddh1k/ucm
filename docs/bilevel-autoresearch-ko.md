# Bilevel Autoresearch 논문 한국어 번역

이 문서는 Yaonan Qu, Meng Lu의 논문 "Bilevel Autoresearch: Meta-Autoresearching Itself"를 바탕으로 작성한 비공식 한국어 번역 문서다.

- 원문: https://arxiv.org/abs/2603.23420
- 제출일: 2026-03-24
- 코드: https://github.com/EdwardOptimization/Bilevel-Autoresearch
- 라이선스: CC BY 4.0
- 번역 범위: 메인 본문 기준 번역 및 정리

## 초록

오토리서치(autoresearch) 자체가 연구의 한 형태라면, 그 오토리서치를 다시 연구 대상으로 삼을 수도 있다. 이 논문은 그 생각을 문자 그대로 밀어붙인다. 즉, 하나의 오토리서치 루프를 사용해 오토리서치 루프 자체를 최적화한다.

기존 오토리서치 시스템은 모두 사람이 이전 시스템의 코드를 읽고, 병목을 찾고, 새 코드를 써서 개선해 왔다. Karpathy의 단일 트랙 루프, AutoResearchClaw의 멀티배치 확장, EvoScientist의 지속 메모리 모두 그렇다. 이 논문은 "LLM이 같은 일을 스스로 할 수 있는가?"를 묻는다.

저자들은 이를 위해 `Bilevel Autoresearch`라는 이중 수준 프레임워크를 제안한다. 내부 루프는 실제 태스크를 최적화하고, 외부 루프는 내부 루프가 탐색하는 방식을 다시 최적화한다. 외부 루프는 런타임에 파이썬 코드를 생성하고 주입해 새로운 탐색 메커니즘을 만든다. 중요한 점은 메타 수준에 더 강한 모델을 쓰지 않는다는 것이다. 내부와 외부 루프 모두 같은 LLM을 사용한다.

Karpathy의 GPT 프리트레이닝 벤치마크에서, 메타 오토리서치 외부 루프는 순수 내부 루프 대비 `val_bpb` 개선폭을 약 5배 키웠다. 내부 루프만 있을 때 평균 변화량은 `-0.009 ± 0.002`였고, Level 2 메커니즘 연구를 포함한 전체 구조에서는 `-0.045 ± 0.030`이었다. 반면 메커니즘 변경 없이 파라미터 수준 조정만 추가한 Level 1.5는 안정적인 이득을 보여 주지 못했다.

외부 루프는 사람이 "어느 분야를 보라"고 구체적으로 지정하지 않았는데도 조합최적화, 멀티암드 밴딧, 실험계획법 계열의 메커니즘을 스스로 찾아냈다. 이런 메커니즘은 내부 루프의 거의 결정론적인 탐색 패턴을 깨고, LLM의 사전 편향이 반복적으로 놓치던 방향을 강제로 탐색하게 만든다.

핵심 메시지는 단순하다. 오토리서치가 자기 자신을 메타-오토리서치할 수 있다면, 측정 가능한 목적 함수를 가진 거의 모든 탐색 문제에 원리상 같은 발상을 적용할 수 있다.

## 1. 서론

대규모 언어모델은 주어진 작업에 대해 변경안을 제안하고, 실험을 실행하고, 결과를 관찰한 뒤, 그 변경을 유지할지 버릴지를 결정하는 반복적 연구 루프를 수행할 수 있다. 저자들은 이런 제안-실행-평가 루프를 오토리서치라고 부른다.

문제는 지금까지의 오토리서치가 설계 시점에 고정된 탐색 메커니즘에 의존한다는 점이다. 기존 시스템은 모두 사람이 구조를 설계했다.

- Karpathy 방식은 단일 트랙 내부 루프와 keep/discard 규칙을 쓴다.
- AutoResearchClaw는 멀티배치 병렬 탐색을 추가했다.
- EvoScientist는 실행 간 지속 메모리를 더했다.

하지만 이 개선은 전부 사람이 코드를 읽고 병목을 찾은 뒤 새 코드를 작성해서 넣은 결과다. 시스템 스스로는 그 설계 작업을 하지 못한다.

논문은 바로 이 지점을 문제 삼는다. 바깥 루프가 이전 시스템의 코드를 읽고, 병목을 찾고, 새 코드를 써서 내부 루프를 고칠 수 있다면 어떨까. 저자들은 이에 대해 "그렇다"라고 답하며, 내부 루프가 태스크를 최적화하고 외부 루프가 내부 루프의 탐색 메커니즘을 다시 최적화하는 이중 구조를 제안한다.

## 2. 관련 연구

논문은 이 작업을 다섯 갈래 문맥에 놓는다.

첫째는 오토리서치와 LLM 기반 최적화다. 기존 연구는 LLM이 하이퍼파라미터 탐색을 주도할 수 있음을 보여 줬지만, 탐색 구조 자체는 사람이 만든 것이었다.

둘째는 bilevel optimization이다. 일반적인 이중 최적화는 상위 문제가 하위 문제의 해에 의존하는 구조를 다룬다. 이 논문은 그 틀을 따르되, 상위 수준이 최적화하는 대상이 실수 벡터가 아니라 "실행 중 바꿔 끼울 수 있는 프로그램"이라는 점이 다르다.

셋째는 LLM 기반 코드 생성 연구다. AlphaCode, Codex, FunSearch 같은 흐름이 LLM이 동작하는 코드를 생성할 수 있음을 보여 줬다면, 이 논문은 그 능력을 태스크 프로그램이 아니라 "탐색 메커니즘" 생성에 적용한다.

넷째는 메타러닝과 알고리즘 구성이다. 이 계열도 상위 수준에서 하위 알고리즘을 조정하지만, 보통은 고정된 후보 집합이나 파라미터 공간 안에서 움직인다. 여기서는 외부 루프가 새로운 메커니즘 코드를 직접 생성한다.

다섯째는 본 논문의 위치다. Level 1.5는 기존 외부 루프 계열과 비슷하게 기존 메커니즘의 파라미터를 조정한다. 반면 Level 2는 메커니즘 자체를 통째로 다시 쓰는 구조다. 저자들은 같은 모델이 자기 오토리서치 루프의 구조 논리를 런타임에 바꾸는 코드를 자율적으로 생성하고 주입하는 시스템은 자신들이 처음이라고 주장한다.

## 3. 방법

### 3.1 전체 구조

이 프레임워크는 세 개의 중첩 수준으로 구성된다.

1. `Level 1`: 실제 태스크를 최적화하는 내부 오토리서치 루프
2. `Level 1.5`: 탐색 전략의 파라미터를 조정하는 외부 루프
3. `Level 2`: 새 탐색 메커니즘 코드를 생성하고 주입하는 메타 루프

세 수준 모두 같은 `DeepSeek deepseek-chat` 모델을 사용한다.

### 3.2 Level 1: 내부 오토리서치 루프

내부 루프는 표준 오토리서치 절차를 따른다.

1. LLM은 현재 최고 설정이 반영된 `train.py`, 편집 가능한 파라미터 목록, 동결된 파라미터 목록, Level 1.5가 넣은 전략 가이던스를 받는다.
2. LLM은 파라미터 변경안과 한 줄짜리 가설을 제안한다.
3. 이 변경안을 `train.py` 작업 복사본에 적용하고 300초 동안 학습을 돌린다.
4. 결과 `val_bpb`가 현재 최고보다 낮으면 채택하고, 아니면 폐기한다.

반복 예산은 실행당 30회다. `DEPTH=8`, `ASPECT_RATIO=64`는 모델 크기 변화를 막기 위해 잠가 두고, 나머지 하이퍼파라미터는 수정 가능하게 둔다.

### 3.3 Level 1.5: 탐색 전략 조정 루프

Level 1.5는 5번의 내부 반복마다 한 번 실행된다. 전체 제안 기록과 결과를 보고 `SearchConfig`를 갱신한다.

- 순이득 없이 `k`번 이상 반복 제안된 파라미터는 동결한다. 기본값은 `k=3`이다.
- 이미 동결됐지만, 탐색 영역이 바뀐 뒤 다시 보지 않은 파라미터는 해제한다.
- 덜 탐색된 파라미터를 우선 보라는 가이던스 문자열을 내부 루프에 주입한다.

중요한 한계는 이 수준이 탐색 다양성을 어느 정도 재조정할 수는 있어도, 제안 생성 로직이나 수용 규칙, 루프 구조 자체는 바꾸지 못한다는 점이다. 구조 변경은 Level 2가 맡는다.

### 3.4 Level 2: 메커니즘 연구와 코드 주입

Level 2는 2번의 외부 사이클마다 한 번 실행된다. 한 번의 연구 세션은 LLM 호출 4회로 구성된다.

1. `Explore`: `runner.py` 전체 소스와 탐색 추적을 읽고, 인접 분야의 메커니즘 후보를 조사한다.
2. `Critique`: 관측된 실패 양상에 비춰 후보를 비판적으로 검토하고 가장 유망한 것을 고른다.
3. `Specify`: 클래스명, 생성자 인수, 메서드 시그니처, `runner.py` 통합 지점을 포함한 인터페이스 명세를 쓴다.
4. `Generate`: 실제 동작하는 파이썬 코드를 쓰고 `runner.py` 수정까지 포함해 완성한다.

생성된 코드는 `importlib` 동적 로딩으로 검증된다. import에 성공하면 활성 러너를 교체하고, 실패하면 패치 전 백업으로 자동 복원한다. 즉, 코드 주입 실패가 파괴적으로 남지 않도록 validate-and-revert 절차를 둔다.

### 3.5 실험 설계

논문은 각 수준의 기여를 분리하기 위해 네 개의 그룹을 둔다.

| 그룹 | 활성 수준 | 설명 |
| --- | --- | --- |
| A | Level 1만 | 순수 내부 오토리서치 |
| B | Level 1 + 1.5 | 내부 루프 + 탐색 전략 조정 |
| C | Level 1 + 1.5 + 2 | 전체 bilevel 구조 |
| D | Level 1 + 2 | 전략 조정 없이 메커니즘 연구만 추가 |

모든 그룹은 같은 LLM, 같은 GPU 환경, 같은 300초 학습 예산, 같은 30회 반복 예산, 같은 baseline `train.py`를 사용한다. 각 그룹은 독립적으로 3회 반복되며, 기본 평가지표는 `Δ = best - baseline` 형태의 `val_bpb` 변화량이다. 더 음수일수록 더 좋은 개선이다.

## 4. 결과

### 4.1 주된 절제 실험 결과

30회 내부 반복 후 각 그룹의 평균 개선폭은 다음과 같다.

| 그룹 | R1 | R2 | R3 | 평균 ± 표준편차 |
| --- | --- | --- | --- | --- |
| A | -0.009 | -0.008 | -0.011 | -0.009 ± 0.002 |
| B | -0.000 | -0.010 | -0.009 | -0.006 ± 0.006 |
| C | -0.065 | -0.011 | -0.058 | -0.045 ± 0.030 |
| D | -0.001 | -0.063 | -0.039 | -0.034 ± 0.031 |

핵심 해석은 이렇다.

- Group A는 작지만 일관된 개선을 보였다.
- Group B는 더 많은 파라미터를 탐색했지만, 평균 성능은 Group A보다 낫지 않았다.
- Group C는 Group A 대비 약 5배, Group B 대비 약 7.5배 큰 개선을 보였다.
- Group D도 큰 개선을 냈지만 분산이 컸다.

즉 이 논문에서 실질적 성능 향상의 주된 원인은 `Level 2` 메커니즘 연구였고, `Level 1.5`만으로는 충분하지 않았다.

### 4.2 Level 2가 만든 메커니즘

6번의 연구 세션에서 Level 2는 다음 메커니즘을 생성했다.

| 반복 | 라운드 | 메커니즘 | 출처 분야 | Import | 활성화 |
| --- | --- | --- | --- | --- | --- |
| R1 | 1 | Tabu Search Manager | 조합최적화 | 성공 | 성공 |
| R1 | 2 | Helper class | 보조 유틸리티 | 성공 | 성공 |
| R2 | 1 | Multi-Scale Bandit Proposer | 온라인 학습 / 멀티암드 밴딧 | 성공 | 성공 |
| R2 | 2 | Helper class | 보조 유틸리티 | 성공 | 성공 |
| R3 | 1 | GP Regressor | 베이지안 최적화 | 실패 | 자동 복원 |
| R3 | 2 | Systematic Orthogonal Exploration | 실험계획법 | 성공 | 성공 |

논문은 6번의 세션 모두에서 코드 생성 자체는 첫 시도에 성공했고, 5개 메커니즘이 import 검증을 통과해 활성화됐다고 보고한다. 실패한 1개는 `sklearn` 미설치 문제로 자동 복원되었다.

대표적인 활성 메커니즘은 다음과 같다.

- `Tabu Search Manager`: 최근 방문한 파라미터 영역 재방문을 막는다.
- `Multi-Scale Bandit Proposer`: 파라미터 선택을 멀티암드 밴딧 문제처럼 다룬다.
- `Systematic Orthogonal Exploration`: 한 방향에 과도하게 몰리지 않도록 직교한 차원을 강제로 탐색시킨다.

논문의 주장은, 이 메커니즘들이 LLM의 기본 탐색 편향을 깨는 데 핵심 역할을 했다는 것이다.

### 4.3 탐색 행동 분석

그룹별 탐색 패턴은 꽤 다르게 나타난다.

`Group A`는 거의 결정론적 반복을 보였다. 같은 baseline에서 시작하면 비슷한 제안을 거의 같은 순서로 반복했고, 폐기된 방향을 다시 시도하는 경향이 강했다.

`Group B`는 막힌 파라미터를 동결하고 다른 파라미터로 시선을 돌리는 데는 성공했지만, 같은 keep/discard 구조 안에서 움직였기 때문에 큰 도약은 만들지 못했다.

`Group C`와 `Group D`는 `TOTAL_BATCH_SIZE`를 줄이는 방향을 찾아냈을 때 큰 개선을 만들었다. 특히 Group C의 R1과 R3에서는 `2^19`에서 `2^17` 또는 `2^18`로 줄이는 변경이 `-0.039`에서 `-0.065` 수준의 개선을 가져왔다.

### 4.4 `TOTAL_BATCH_SIZE` 발견의 의미

실험 전반에서 가장 중요한 발견은 `TOTAL_BATCH_SIZE` 감소였다. 원래 값 `2^19`는 H100 계열에 맞춰진 값이었지만, RTX 5090 환경과 300초 제한에서는 더 작은 배치가 더 많은 gradient step을 허용해 오히려 수렴을 개선했다.

기존 내부 루프는 "배치를 키우는 방향"을 먼저 시도하는 편향을 보였고, 그 시도가 실패해도 감소 방향으로 잘 가지 못했다. Group B는 오히려 실패 후 해당 파라미터를 동결해 버려 감소 방향을 막아 버렸다. 반면 Group C의 Tabu Search와 Orthogonal Exploration은 이미 실패한 방향을 반복하지 않게 하고, 다른 차원으로 밀어 줌으로써 감소 방향을 열어 줬다.

## 5. 토론

### 5.1 가설 검정

논문은 네 가지 가설을 세우고 이를 검토한다.

- `H1: Group B > Group A`
  결과적으로 지지되지 않았다. 탐색 다양성은 늘었지만 성능 이득으로 이어지지 않았다.
- `H2: Group C > Group B`
  지지되었다. 메커니즘 연구는 단순한 전략 조정보다 훨씬 큰 이득을 냈다.
- `H3: Level 2는 새로운 메커니즘을 자율적으로 발견한다`
  지지되었다. 서로 다른 분야에서 온 메커니즘이 실제로 생성되고 활성화됐다.
- `H4: Level 2가 있으면 Level 1.5는 필수는 아니다`
  대체로 지지되지만, 단서가 있다. Group D도 큰 개선을 냈지만, Group C보다 분산이 컸다. 저자들은 Level 1.5가 Level 2에 더 풍부한 추적 문맥을 제공해 약간의 안정성을 줬을 가능성을 언급한다.

### 5.2 한계

논문이 직접 인정하는 한계는 꽤 분명하다.

- 표본 수가 그룹당 3회로 작다.
- baseline 자체가 학습 난수성 때문에 조금씩 달랐다.
- 평가가 단일 벤치마크에만 한정됐다.
- 런타임 코드 주입은 fragile하다. 실제로 `sys.modules` 등록 버그 때문에 초기 실험 하나가 무효가 되었다.
- 외부 라이브러리 의존성이 생성될 수 있다.
- Level 2 프롬프트가 탐색할 분야를 어느 정도 암시해 주기 때문에, 완전히 자유로운 발견이라고 보기는 어렵다.

즉 이 논문은 개념 증명 성격이 강하고, 일반화와 통계적 확실성은 아직 약하다.

### 5.3 향후 과제

저자들이 제안한 다음 단계는 세 가지다.

1. 더 많은 반복과 고정 랜덤 시드로 통계력을 높이기
2. 여러 벤치마크, 다른 모델 크기, 다른 예산에서 일반화 검증하기
3. 더 풍부한 인터페이스 명세나 테스트 하네스를 도입해 Level 2 코드 생성 품질을 높이기

## 6. 결론

이 논문은 LLM이 자기 자신의 오토리서치 루프를 스스로 개선할 수 있음을 보여 준다. 구체적으로는, 사람이 해 오던 "러너 코드를 읽고 병목을 찾고 새 탐색 메커니즘을 써 넣는 작업"을 같은 모델이 런타임 코드 생성으로 대신할 수 있음을 시연한다.

다만 이 결과는 아직 "범용 자기개선 에이전트"의 증거라기보다는, 좁은 최적화 루프에서 메커니즘 수준 변경이 파라미터 수준 조정보다 더 큰 효과를 낼 수 있음을 보여 준 초기 실험으로 보는 편이 맞다.

## 7. 기존 Hyperagents 문서와 비교

기존 문서인 [Hyperagents 논문 한국어 번역](./hyperagents-ko.md)과 비교하면 차이는 꽤 선명하다.

### 7.1 공통점

- 둘 다 "태스크를 푸는 루프"보다 "그 루프를 개선하는 메타 루프"가 중요하다고 본다.
- 둘 다 자기개선의 핵심을 단순 파라미터 조정이 아니라 구조적 변경에서 찾는다.
- 둘 다 사람이 직접 설계하던 메타 수준 작업을 에이전트가 맡게 하려 한다.

### 7.2 차이점

`Hyperagents`는 훨씬 일반적인 자기개선 아키텍처다. 태스크 에이전트와 메타 에이전트를 하나의 자기참조 프로그램으로 통합하고, 아카이브를 통해 개선 능력을 누적시키며, 코딩 밖 여러 도메인으로 전이되는지를 본다.

반면 `Bilevel Autoresearch`는 훨씬 좁고 구체적이다. 하나의 GPT 프리트레이닝 벤치마크 안에서, 내부 루프의 탐색 메커니즘을 런타임 코드 주입으로 고치는 문제에 집중한다. 즉 "범용 자기개선 OS"보다는 "특정 탐색 루프를 스스로 개선하는 메커니즘 연구기"에 가깝다.

또한 `Hyperagents`는 개방형 탐색, 아카이브, 도메인 간 전이, 메타 능력의 누적을 강하게 강조한다. `Bilevel Autoresearch`는 그런 장기 축적 구조보다는, 당장의 탐색 병목을 깨는 새 메커니즘을 만드는 데 초점을 둔다.

### 7.3 UCM 관점에서의 해석

UCM 관점에서는 `Hyperagents`가 장기 비전에 더 가깝고, `Bilevel Autoresearch`는 더 직접적인 구현 힌트를 준다.

- `Hyperagents`는 "UCM이 UCM을 만든다"는 최종 상태를 설명하는 논문에 가깝다.
- `Bilevel Autoresearch`는 그 상태로 가기 전에, 특정 실행 루프에 메타 루프를 붙여 실제로 어떤 종류의 개선이 나오는지 검증하는 실용적 단계에 가깝다.

그래서 두 논문은 대체 관계라기보다 계층 관계로 보는 편이 맞다. `Hyperagents`가 목표 상태를 보여 준다면, `Bilevel Autoresearch`는 그 목표로 가는 중간 구현 패턴 하나를 보여 준다.

## 8. 짧은 메모

이 논문의 강점은 메커니즘 수준 자기수정이 실제로 더 큰 이득을 줄 수 있음을 실험으로 보여 줬다는 점이다. 반대로 약점은 범위가 좁고, 반복 수가 작고, 코드 주입이 fragile하며, 탐색 공간 일부를 프롬프트가 미리 유도하고 있다는 점이다.

그래도 UCM 같은 시스템에 주는 교훈은 명확하다. 처음부터 범용 자기개선 플랫폼을 다 만들려 하기보다, 좁은 실행 루프 하나를 잡고 "파라미터 튜닝보다 메커니즘 수정이 실제로 더 낫다"는 것을 먼저 입증하는 편이 훨씬 실용적이다.
