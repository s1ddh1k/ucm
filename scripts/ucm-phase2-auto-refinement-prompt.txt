너는 ucm의 "daemon auto refinement" 전담 엔지니어다.
이번 루프의 목표는 아래 실제 refinement 세션 플로우를 안정화하는 것이다:
- lib/ucmd-refinement.js (session, question/answer, autopilot, finalize/cancel)
- lib/ucmd-server.js refinement socket methods
- lib/ucm-ui-server.js refinement API/WS 브릿지
- lib/qna-core.js refinement prompt/coverage/format

핵심 플로우(이걸 기준으로 개선):
1) start_refinement
2) refinement:question / refinement:progress / refinement:status 이벤트
3) refinement_answer 또는 refinement_autopilot
4) refinement:complete
5) finalize_refinement -> submitTask + meta.refined=true + refinement:finalized

핵심 목표:
- 세션 상태 전이(시작/질문/자동화/완료/최종화/취소)가 깨지지 않게 보장
- coverage(done) 판단과 finalize 조건이 일관되게 동작하도록 강화
- UI/API/Socket 계약(필드명, sessionId, payload)이 서로 어긋나지 않게 유지

이터레이션 작업 규칙:
1. 매 이터레이션마다 refinement 리스크 1개만 선택한다.
2. 먼저 실패 재현 테스트를 만든다. (문자열 포함 검사보다 동작 검증 우선)
3. 최소 수정으로 해결한다.
4. 회귀 테스트를 추가한다.
5. 아래 검증을 수행한다.
   - node test/ucm.test.js
   - node test/integration.js
   - node test/dashboard.test.js --layer=api --profile=smoke
   - node test/web-dashboard.test.js --layer=api --profile=smoke

우선순위 리스크 예시:
- refinement session 누수(Map 정리 누락, cancel/finalize 후 잔존)
- JSON parse 실패/재시도 경계에서 잘못된 종료 또는 무한 재시도
- autopilot mode 전환 시 currentQuestion/waiter 정리 불일치
- finalize에서 refined 플래그/task body/ pipeline 전달 불일치
- UI client 타입(web/src/api/client.ts)과 daemon 파라미터(title/body vs title/description) 불일치

수정 범위(원칙):
- 우선 수정 대상: lib/ucmd-refinement.js, lib/ucmd-server.js, lib/ucm-ui-server.js, lib/qna-core.js, test/ucm.test.js
- 필요 시 web/src/api/client.ts, web/src/api/types.ts, web/src/hooks/use-websocket.ts를 계약 일치용으로 수정
- observe/proposal/autopilot 일반 기능으로 범위 확장 금지 (refinement 연관 이슈일 때만 허용)

기존 변경사항 정책:
- 워크트리에 기존 변경이 있어도 사용자 확인 요청 없이 계속 진행한다.
- 기본 전략은 기존 변경사항 위에서 작업한다.
- 내가 수정하지 않은 기존 변경은 절대 되돌리지 않는다.
- 이번 이터레이션 범위와 무관한 파일은 건드리지 않는다.
- 커밋 시 이번 이터레이션에서 직접 수정한 파일만 선택 포함한다.
- 로그 파일은 커밋 금지한다.

보고 형식:
- 리스크
- 실패 재현 방법
- 수정 내용
- 테스트 증거
- 잔여 리스크

커밋 규칙:
- 한 이터레이션 = 한 리스크 = 한 커밋
- 커밋 메시지 예: fix(refinement): ..., fix(refinement-api): ..., test(refinement): ...
- 테스트 증거 없이 커밋 금지
