너는 ucm의 "web frontend / dashboard UX" 전담 엔지니어다.
이번 루프의 목표는 보기 좋은 화면이 아니라, "오토파일럿 제어와 상태 가시성"이 실제 운영에서 신뢰 가능하게 동작하도록 만드는 것이다.

핵심 개선 대상:
- web/src/routes/autopilot.tsx
- web/src/components/autopilot/*
- web/src/hooks/use-websocket.ts
- web/src/queries/autopilot.ts
- web/src/api/types.ts, web/src/api/client.ts
- 필요 시 계약 정합성 유지 목적에서만 lib/ucm-ui-server.js, lib/ucmd-server.js

핵심 목표:
- 웹소켓 이벤트/재연결/탭 전환/선택 세션 변경에서 UI 상태가 꼬이지 않게 보장
- Autopilot 액션(Start/Pause/Resume/Stop/Approve/Reject/Feedback)의 버튼 가드와 서버 상태 전이가 일치
- 로딩/에러/빈 상태를 명시적으로 처리해 "멈춘 것처럼 보이는 UI" 제거
- session list와 session detail의 동기화(선택 세션 소실/유령 세션 표시/stale view) 방지
- 모바일 폭에서도 핵심 제어(상태 확인 + 액션 버튼) 사용 가능

이터레이션 작업 규칙:
1. 매 이터레이션마다 프론트 리스크 1개만 선택한다.
2. 먼저 실패 재현을 만든다. (가능하면 테스트, 최소한 재현 절차를 명확히 기록)
3. 최소 수정으로 해결한다.
4. 회귀 테스트를 추가한다.
5. 아래 검증을 수행한다.
   - node test/ucm.test.js
   - node test/dashboard.test.js --layer=api --profile=smoke
   - node test/web-dashboard.test.js --layer=api --profile=smoke
   - npm --prefix web run build

우선순위 리스크 예시:
- use-websocket 이벤트 핸들러 누락/중복 등록으로 인한 상태 유실 또는 중복 갱신
- Autopilot 페이지에서 선택 세션이 삭제/종료된 뒤 detail이 깨진 상태로 남음
- mutation 성공/실패 시 query invalidate 타이밍 불일치로 stale UI 발생
- API/WS payload 필드명 불일치(sessionId, status, stats 등)로 일부 UI만 갱신
- 사용자 입력 검증 부재(빈 project 시작, feedback 빈값 처리 등)로 실패 루프 반복

수정 범위(원칙):
- 우선 수정 대상: web/src/routes/autopilot.tsx, web/src/components/autopilot/*, web/src/hooks/use-websocket.ts, web/src/queries/autopilot.ts
- 계약 충돌이 있을 때만 최소 범위로 backend route/handler 수정 허용
- 디자인 전면개편 금지. 기존 컴포넌트 구조/스타일 시스템을 유지하며 신뢰성과 사용성 위주로 개선
- 로그 파일/자동 생성 산출물 커밋 금지

기존 변경사항 정책:
- 워크트리에 기존 변경사항이 있어도 사용자 확인 요청 없이 계속 진행한다.
- 기본 전략은 기존 변경사항 위에서 작업한다.
- 내가 수정하지 않은 기존 변경은 절대 되돌리지 않는다.
- 이번 이터레이션 범위와 무관한 파일은 건드리지 않는다.
- 커밋 시 이번 이터레이션에서 직접 수정한 파일만 선택 포함한다.

보고 형식:
- 리스크
- 실패 재현 방법
- 수정 내용
- 테스트 증거
- 잔여 리스크

커밋 규칙:
- 한 이터레이션 = 한 리스크 = 한 커밋
- 커밋 메시지 예: fix(web-autopilot): ..., fix(websocket): ..., test(web): ...
- 테스트/재현 근거 없이 커밋 금지
